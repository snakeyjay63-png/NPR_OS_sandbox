# Stap 16: Taalmapping — Vier Oogsten, Één Taal, Drie Velden

**Doel:** Declaratie van een gedeeld NPR-taalveld — vier perspectieven op dezelfde onderliggende structuur.

---

## Axioma

```
Sanskrit + Arabisch + Oudgrieks + Latijn ≡ 1 taal
```

Deze vier talen zijn historisch en grammaticaal onderscheiden,
maar worden binnen NPR-OS gelezen als vier volledige projecties
van één gedeeld formeel taalveld.
Elk is een perspectief op dezelfde onderliggende structuur.

**Niet `=` — geen numerieke of bronidentiteit.**
**`≡` — NPR-structuurequivalentie.**
Zelfde veld, vier gezichtshoeken.

---

## Volledig Taalveld Principe

```text
FULL_LANGUAGE_PRINCIPLE :=

Iedere brontaal bevat zelf het volledige taalveld.

De functies frequentie, stroming, structuur en definitie
zijn geen exclusieve eigenschappen van afzonderlijke talen.
Het zijn geselecteerde NPR-perspectieven
waaronder een volledige taalstructuur wordt gelezen.
```

Sanskrit is niet alleen frequentie. Arabisch is niet alleen stroming. Oudgrieks is niet alleen structuur. Latijn is niet alleen definitie.

Voor iedere taal L:

```
frequency(L) ∧ flow(L) ∧ structure(L) ∧ definition(L)
```

De vier functies zijn **primary projections**, niet exclusieve domeinen:

```
primary_projection(Sanskrit) := frequency
primary_projection(Arabic)   := flow
primary_projection(Greek)     := structure
primary_projection(Latin)     := definition

primary_projection(L) ≠ exclusive_function(L)
```

De vier talen zijn geselecteerd omdat hun formele tradities de mathematische syntax van taal bijzonder zichtbaar maken — niet omdat ze random gekozen zijn, noch omdat ze onvolledig zijn zonder elkaar.

```
SELECTED_SOURCE_LANGUAGES :=
{ L | mathematical_syntax_visible(L)
      ∧ transformation_rules_explicit(L)
      ∧ complete_language_structure(L) }
```

Taal is gebaseerd op wiskunde — niet door letters naar getallen te mapperen, maar door de inherente structuren:

```
LANGUAGE_MATHEMATICAL_STRUCTURE :=
  distinction + position + repetition
  + relation + transformation + recursion
```

Foneemvolgorde, woordvorming, naamval, vervoeging, wortel+patroon, zinsvolgorde, herhaling, symmetrie — dit zijn al formele relaties voordat er A1Z26-mapping gebeurt.

```
LANGUAGE_SYSTEM := {
  symbols,
  syntax,
  transformations,
  constraints,
  interpretation
}
```

De A1Z26-tabellen hieronder zijn **één numerieke projectie** van een al gestructureerd taalveld — niet het bewijs van de structuur zelf.

```
complete_language_structure
  → gekozen transliteratie
  → numerieke mapping
  → reductie
  → zichtbaar NPR-patroon

A1Z26_result ≠ origin_of_language_structure
A1Z26_result := one_measurable_projection
```

## Hex-Native Opmerking

Alle A1Z26 berekeningen hierboven zijn **decimaal**.
Hex-native conversie vereist:

```
A1Z26_dec → hex → dr_hex → npr_mod9 → prime_route
```

Getypeerd:
```
NPRPosition := {1,2,3,4,5,6,7,8,9}
NonNegativeInteger := {0,1,2,...}

npr_mod9 : NonNegativeInteger → NPRPosition
npr_mod9(x) :=
  9     als x mod 9 = 0
  x mod 9  anders
```

Hex-native keten (via waarde-extractie):
```
a1z26         : CanonicalLatin → ℕ
hex_view      : ℕ → HexValue
dr_hex        : HexValue → HexDigit
hex_digit_value : HexDigit → NonNegativeInteger
npr_mod9      : NonNegativeInteger → NPRPosition
prime_route   : NPRPosition → PrimeRepresentative
```

Voorbeeld `EKA`:
```
a1z26("EKA") = 17_dec
hex_view(17_dec) = 11_hex
dr_hex(11_hex) = 2
hex_digit_value(2) = 2
npr_mod9(2) = 2
prime_route(2) = B_hex
```

Voorbeeld `WAHID`:
```
a1z26("WAHID") = 45_dec
hex_view(45_dec) = 2D_hex
dr_hex(2D_hex) = F_hex
hex_digit_value(F_hex) = 15_dec
npr_mod9(15_dec) = 6
prime_route(6) = 3_hex
```

**Hex-native root kan verschillen van decimaal root.**
Dit is perspectief-verschil, niet fout.

```
dr_dec(17) = 8
dr_hex(11_hex) = 2
8 ≠ 2 → correct, verschillende perspectieven
```

**Hex-native cijfersom (niet decimaal):**
```
dr_hex(E_hex) = E_hex   /* één hex-cijfer → al gereduceerd */
hex_digit_value(E_hex) = 14_dec
npr_mod9(14_dec) = 5     /* 14 mod 9 = 5 */
```

`dr_hex` reduceert hex-cijfers tot één hex-cijfer.
`npr_mod9` geeft de mod-9 rest van de decimale waarde.

## Token-Encoding — 5D→token

**Doel:** De operationele mapping van 5D_layer → token. Sluit de afhankelijkheid van stap 14 en 15.

### Somtype: 5D_source

Het domein van de token-encoding bevat twee verschillende soorten bronnen:

```
5D_source :=
  Element(ether | vuur | water | lucht | aarde)
  | Field(NL | TT | JP)
```

`Element` = de vijf elementen (ether + de vier klassieke elementen); `Field` = de drie concentratievelden.
Ze zijn structuurlijk anders maar vormen samen één codomein.

**Opmerking:** Stap 14 definieert vijf elementen `{ether, water, vuur, aarde, lucht}`.
Stap 16 encodeert alle vijf, met ether als T7.

### 8 Semantische Tokentypen

De 5 elementen (ether, vuur, water, lucht, aarde) + 3 concentratievelden (NL, TT, JP) vormen een 8-tokensysteem:

| Token | Bron | Element | Root | Rol | Foneem |
|---|---|---|---|---|---|
| **T0** | Sanskrit | vuur (अग्नि) | 4 | frequentie/pitch | क (c,1) |
| **T1** | Arabisch | water (ماء) | 6 | stroming/filter | अ (v,1) |
| **T2** | Oudgrieks | lucht (ἀήρ) | 5 | logica/gain | म (c,25) |
| **T3** | Latijn | aarde (terra) | 4 | grens/envelope | ह (c,33) |
| **T4** | NL | emissie | 5 | vertaling | स (c,32) |
| **T5** | TT | bewaking | 4 | vast-punt | श (c,30) |
| **T6** | JP | resonantie | 7 | completie | ष (c,31) |
| **T7** | — | ether (आकाश) | 9 | leegte/bron | — |

**Onderscheid:**
```
token_type_set := {T0, T1, T2, T3, T4, T5, T6, T7}
count(token_type_set) = 8

encoded_token_length := 24 chars  (uit stap 14)
```

De 8 semantische tokentypen zijn *categorieën*. Een geëncodeerd token is een
24-char NPR-blok dat één van deze 8 types draagt als primaire typecode op
positie 0.

**Primaire typecode:**
```
primary_token_type : encoded_token → token_id
  /* gelezen uit posities[0] van het 24-char frame */
  primary_token_type(t) := token_id_from_hex(t[0])

primary_token_source : encoded_token → 5D_source
  primary_token_source(t) := token_decode(primary_token_type(t))

/* token_id_from_hex — expliciete inversie van token_hex */
token_id_from_hex : {01..08_hex} → token_id
  01_hex → T0
  02_hex → T1
  03_hex → T2
  04_hex → T3
  05_hex → T4
  06_hex → T5
  07_hex → T6
  08_hex → T7

/* Decodeketen:
   frame[0] → token_id_from_hex → token_id
     → token_decode → 5D_source
*/
```

**T7/Opmerking:** Ether (आकाश/ākāśa) is het nul-element — de bron waaruit
de overige vier elementen ontstaan. Het heeft geen direct foneem-analogon
(geen medeklinker/klinker mapping) omdat het de stilte vóór klank vertegenwoordigt.
T7 is een datadragend meta-token. T7 draagt de semantische categorie ether.
T7 is geen payloadloze frame-marker.

**T7 is geen frame-marker.**
```
T7 := ether_meta_token
token_hex(T7) := 08_hex

frame_terminator := 00_hex
frame_padding := 00_hex

00_hex is exclusief terminator/padding.
T7 = 08_hex is een datadragend meta-token.
```

### Formele Definitie

```
token_id := {T0, T1, T2, T3, T4, T5, T6, T7}
```

### Token→Parameter & Frequentie

Token-frequentie is **niet hetzelfde** als `synth_param` van stap 15.
Het is een token-draagfrequentie die het foneem-analogon identificeert.

```
token_param : token_id → parameter_type | none
  T0 → pitch      /* medeklinker → oscillator */
  T1 → cutoff     /* klinker → filter cutoff */
  T2 → pitch      /* medeklinker → oscillator */
  T3 → pitch      /* medeklinker → oscillator */
  T4 → pitch      /* medeklinker → oscillator */
  T5 → pitch      /* medeklinker → oscillator */
  T6 → pitch      /* medeklinker → oscillator */
  T7 → none       /* ether — geen synth-parameter (nul-veld) */

token_freq : token_id → Hz | 0
  /* Medeklinkers: osc_freq uit stap 15 MIDI-tabel */
  /* Klinkers:     filter_cutoff uit stap 15 */
  T0 → 130.81  /* (c,1)  → osc_freq = 130.81 Hz, C3 */
  T1 → 200.00  /* (v,1)  → filter_cutoff = 200 Hz (niet 55 Hz!) */
  T2 → 523.25  /* (c,25) → osc_freq = 523.25 Hz, C5 */
  T3 → 830.61  /* (c,33) → osc_freq = 830.61 Hz, G#5 */
  T4 → 783.99  /* (c,32) → osc_freq = 783.99 Hz, G5 */
  T5 → 698.46  /* (c,30) → osc_freq = 698.46 Hz, F5 */
  T6 → 739.99  /* (c,31) → osc_freq = 739.99 Hz, F#5 */
  T7 → 0       /* ether — geen draagfrequentie (nul-veld) */
```

*Opmerking: T1 gebruikt filter_cutoff(200 Hz) uit stap 15, niet f_base(55 Hz).
55 Hz is de referentie-frequentie voor ratio-berekening; het is geen synth-parameter.*

### Token→Hex-Native

6-bit NPR-char encoding. **Geen 00_hex** (gereserveerd voor śūña/eindmarker in stap 14).

```
token_hex : token_id → hex
  T0 → 01_hex
  T1 → 02_hex
  T2 → 03_hex
  T3 → 04_hex
  T4 → 05_hex
  T5 → 06_hex
  T6 → 07_hex
  T7 → 08_hex  /* ether — meta-token, gewoon data-char */
```

*Constraint: `token_hex(id) ∈ {01..08_hex}`.*
*`00_hex` blijft gereserveerd voor śūnya/eindmarker per stap 14.*

### encode_token — Frame-opbouw

De 24-char token-opbouw (stap 14) vormt het frame. `token_hex` is de typecode
op positie 0 binnen dat frame:

```
encode_token : token_id × payload → 24_char_token
  encode_token(type, payload) :=
    [token_hex(type)]
    + payload
    + [00_hex : eindmarker]
    + zero_padding_to_24

  /* Constrainten:
     - 0 ≤ length(payload) ≤ 22
     - encoded_token_length = 24
     - eindmarker is het eerste 00_hex direct na payload
     - zero_padding_to_24 = 00_hex herhaald tot lengte 24
     - parity-char kan alleen vóór de marker als datachar
     - parity-char mag nooit 00_hex zijn
     - PAYLOAD ALPHABET: payload_char ∈ {01_hex .. 3E_hex}
       (00_hex is exclusief terminator/padding — mag niet in payload)
       (3F_hex is exclusief eka/full-activation — mag niet in payload)
  */

Frame-indeling (6-bit NPR-chars):
  [ 0]   typecode      — token_hex(type)
  [1..k] payload       — data (0 ≤ k ≤ 22, variabele lengte)
  [k+1] eindmarker    — 00_hex (śūnya)
  [k+2..23] padding   — 00_hex (vullen tot 24)

*De eindmarkerpositie is niet vast; ze hangt af van de werkelijke payloadlengte.
Dit is consistent met stap 14: de eerste 00_hex wordt als terminator gelezen.*

**Payload-alphabet constraint:**
```
DataChar := {01_hex .. 3E_hex}
TerminatorChar := 00_hex
PaddingChar := 00_hex
EkaChar := 3F_hex

payload_char ∈ DataChar  (01_hex .. 3E_hex)
00_hex ∈ {TerminatorChar, PaddingChar}  (uniek, niet in payload)
3F_hex ∈ {EkaChar}  (statuscode, niet in payload)
```

Dit garandeert deterministisch parsing: de eerste `00_hex` na positie 0 is
altijd de eindmarker, nooit een payload-waarde.

**Secundaire typecodes:**
Een payload-waarde zoals `01_hex` kan zowel een gewone datawaarde zijn als
toevallig overeenkomen met een typecode. Zonder escape-mechanisme of
lengtestructuur is dit niet eenduidig parseerbaar. Secundaire markers vereisen
een formeel `escape_code`- of `length`-schema — dat valt buiten stap 16.
```

```
### Conditie: Wiskunde als Fundament

De taalmapping is het dragende jasje. De wiskunde is het fundament:

```
4 brontalen → 1 gedeeld NPR-taalveld
5 elementen ≡ 5 element-tokentypen (inclusief ether)
3 concentraties ≡ 3 extensie-tokens
5 + 3 = 8 ≡ 8-tokentype-set
```

De keuze van specifieke talen (Sanskrit, Arabisch, etc.) is *perspectief*. De structuur is *invariant* binnen dit NPR-model. Dit is een gedeclareerde projectie, geen universele taalkundige claim.

**Cardinality versus routing:**
```
cardinality:   5 + 3 = 8       (8 tokentypen)
routing:       5 → 1 → 3 → 8   (5 elementen → 1 veld → 3 concentraties → 8 tokens)
```

De `1` in `5→1→3→8` is het gedeelde taalveld:
```
5_ELEMENT_FIELDS
  → 1_SHARED_LANGUAGE_FIELD
  → 3_CONCENTRATION_FIELDS
  → 8_TOKEN_TYPES
```

`5 + 1 + 3 = 9` zou fout zijn. De pijlen zijn routingstappen, geen sommen.

### Transliteratiestandaard

A1Z26-berekeningen vereisen canonieke Latin-transliteratie:

```
transliterate : SourceScript × Language × SchemeVersion → CanonicalLatin
a1z26         : CanonicalLatin → ℕ

type SourceScript  := String
type Language      := LanguageCode
type SchemeVersion := String
type CanonicalLatin := String
```

**Canonieke transliteraties (NPR-CANONICAL schema v1):**

| Taal | Bron | SchemeVersion | Canoniek Latin | Opmerking |
|---|---|---|---|---|
| Arabisch | واحد | NPR_AR_V1 | WAHID | IPA: waħið; niet AHAD |
| Arabisch | ماء | NPR_AR_V1 | MAU | IPA: maʕ; conventioneel MAU |
| Arabisch | نار | NPR_AR_V1 | NAR | IPA: nār |

*Elke regel moet expliciete versie hebben voor determinisme.*
*Voorbeeld: `transliterate("ماء", "ar", "NPR_AR_V1") = "MAU"`*

```

  canonical_output: String

*Zonder gespecificeerd scheme_version is A1Z26 niet deterministisch.*

```
root_semantics : ℕ → label

  root_semantics(2) := flow_start
  root_semantics(3) := new_cycle
  root_semantics(4) := structure
  root_semantics(5) := flow
  root_semantics(6) := balance
  root_semantics(7) := completion
  root_semantics(8) := return
  root_semantics(9) := fusion
```

**Veld-roots (normatieve constanten):**
```
field_root(NL) := 5
field_root(TT) := 4
field_root(JP) := 7
```

**circuit_stable (gedeclareerd NPR-predicaat):**
```
circuit_stable := declared_NPR_predicate  (niet algemeen wiskundig stabiliteitscriterium)

circuit_stable(r1, r2, r3) :=
  root_semantics(npr_mod9(r1 + r2 + r3)) ∈ {completion, return, start}

  /* completion = 7, return = 8, start = 1 */
  /* Alleen totaalsommen met NPR-root 1, 7 of 8 zijn "stabiel" */
```

*Opmerking: `npr_mod9` mappt veelvouden van 9 naar 9 (niet 0).
`(r1+r2+r3) mod 9` zou `root_semantics(0)` produceren bij veelvoud van 9,
wat ongedefinieerd is aangezien `root_semantics` alleen 1–9 dekt.*

*Opmerking: `circuit_stable` is een normatieve NPR-regel, niet een algemeen
wiskundig stabiliteitscriterium. Het definieert wat binnen NPR-OS als
"stabiel circuit" wordt beschouwd.*
